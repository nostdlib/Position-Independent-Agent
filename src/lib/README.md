# Libraries

Cryptography, TLS 1.3, HTTP, WebSocket, DNS-over-HTTPS, and JPEG encoding — implemented from scratch with no external dependencies.

## Cryptography

### SHA-2: Traits-Based Template Design

**File:** `crypto/sha2.h`

A single `SHABase<Traits>` template implements both SHA-256 and SHA-384:

| Trait | SHA-256 | SHA-384 |
|---|---|---|
| Word size | 32-bit | 64-bit |
| Rounds | 64 | 80 |
| Block size | 64 bytes | 128 bytes |
| Digest size | 32 bytes | 48 bytes |

Round constants (K) and initial hash values (H0) are generated by **static methods** rather than stored as global arrays — keeping them out of `.rdata`. The compiler embeds them as immediate values in the `.text` section.

Big-endian word packing (`Pack`/`Unpack`) converts between byte arrays and native word types. Rotation functions (`Sigma0`, `Sigma1`, `sigma0`, `sigma1`) compile to single `ROR` instructions via the bit rotation pattern in `bitops.h`.

Supports both one-shot `Hash()` and incremental `Update()`/`Final()` for streaming data.

**HMAC** wraps SHA with standard ipad/opad key padding (RFC 2104).

### ChaCha20-Poly1305: No Lookup Tables, No Timing Leaks

**File:** `crypto/chacha20.h`

The TLS 1.3 cipher suite, fully branchless:

**ChaCha20** (stream cipher):
- 256-bit key, 96-bit nonce, 32-bit counter
- 20 rounds of quarter-round operations on a 4×4 matrix of 32-bit words
- Each quarter-round: add, XOR, rotate — no S-boxes, no lookup tables
- Generates a 64-byte keystream block per counter value

**Poly1305** (one-time MAC):
- 26-bit limb arithmetic on a 5-limb accumulator (130-bit intermediate values)
- Key clamping: clear top 4 bits and bottom 2 bits of each 32-bit chunk of `r`
- Fully branchless arithmetic — constant-time regardless of input, preventing cache timing side channels
- The tag is `((accumulator * r) mod (2^130 - 5)) + s` where `s` is the second half of the one-time key

**AEAD construction**: Encrypts plaintext, authenticates both associated data and ciphertext, produces a 16-byte tag.

Stack-only, non-copyable, non-movable — enforces one-time key usage.

### Elliptic Curve Cryptography: Constant-Time ECDH

**File:** `crypto/ecc.h`

Full ECC implementation for TLS 1.3 key exchange:

- **Curves**: secp256r1 (P-256) and secp384r1 (P-384)
- **Coordinate system**: Jacobian projective coordinates — point addition and doubling avoid per-operation modular inverse (expensive), deferring to a single inversion at the end
- **Scalar multiplication**: Montgomery ladder — processes every bit of the scalar regardless of value, making execution time constant (defeats simple power analysis)
- **Variable-length integer (VLI) arithmetic**: Fixed-size arrays of 64-bit words with compile-time size selection per curve
- **Curve-specific fast reduction**: Exploits the special form of NIST primes (e.g., P-256 = 2^256 - 2^224 + 2^192 + 2^96 - 1) for modular reduction faster than generic Barrett reduction
- **Point validation**: Verifies imported public keys satisfy the curve equation before use (prevents invalid-curve attacks)

## TLS 1.3: Full Handshake Implementation

**File:** `network/tls/tls_client.h`, `tls_cipher.h`, `tls_hkdf.h`

Complete TLS 1.3 client (RFC 8446):

### Handshake Flow

```
Client                                        Server
  │                                             │
  ├─ ClientHello ─────────────────────────────►│
  │  (random, cipher_suites,                    │
  │   key_share: ECDH public key,              │
  │   supported_versions: 0x0304)               │
  │                                             │
  │◄───────────────────────── ServerHello ──────┤
  │  (random, cipher_suite,                     │
  │   key_share: ECDH public key)               │
  │                                             │
  │  ── derive handshake keys ──                │
  │                                             │
  │◄──────────── EncryptedExtensions ──────────┤
  │◄──────────── Certificate ──────────────────┤
  │◄──────────── CertificateVerify ────────────┤
  │◄──────────── ServerFinished ───────────────┤
  │                                             │
  │  ── derive application keys ──              │
  │                                             │
  ├─ ClientFinished ──────────────────────────►│
  │                                             │
  │◄═══════════ Application Data ══════════════╡
```

### Key Schedule (HKDF)

TLS 1.3 derives all keys through HKDF (RFC 5869):

```
ECDH shared secret
  │
  ├─ HKDF-Extract(salt=0, IKM=shared_secret) → handshake_secret
  │    ├─ Derive-Secret("c hs traffic") → client_handshake_traffic_secret
  │    └─ Derive-Secret("s hs traffic") → server_handshake_traffic_secret
  │
  └─ HKDF-Extract(salt=derived, IKM=0) → master_secret
       ├─ Derive-Secret("c ap traffic") → client_application_traffic_secret
       └─ Derive-Secret("s ap traffic") → server_application_traffic_secret

Each traffic secret → HKDF-Expand-Label("key") → encryption key
                    → HKDF-Expand-Label("iv")  → nonce/IV
```

### Channel Buffer

TLS records may arrive fragmented or interleaved. The channel buffer reassembles partial records before passing complete messages to the state machine.

## DNS-over-HTTPS: Encrypted Name Resolution

**File:** `network/dns/dns_client.h`

DNS queries sent over HTTPS (RFC 8484) to prevent DNS snooping:

```
1. Build DNS wire-format query (RFC 1035): QNAME + QTYPE + QCLASS
2. HTTP POST to https://1.1.1.1/dns-query
   Content-Type: application/dns-message (binary, not base64)
3. Parse DNS response: extract A or AAAA record
4. Fallback: if Cloudflare fails, try Google (8.8.8.8)
5. Fallback: if AAAA fails, retry as A query
```

Short-circuits `"localhost"` → `127.0.0.1` without network access.

## WebSocket: RFC 6455

**File:** `network/websocket/websocket_client.h`

Full WebSocket protocol implementation:

### Client-to-Server Masking

All client frames are XOR-masked with a 32-bit random key (RFC 6455 requirement):

```c
for (int i = 0; i < payload_len; i++)
    payload[i] ^= mask_key[i % 4];
```

### Variable-Length Encoding

Frame payload length uses three encoding modes:
- 0-125: 7-bit length in the header byte
- 126-65535: 16-bit extended length
- 65536+: 64-bit extended length

### Control Frame Handling

Ping frames trigger automatic Pong responses. Close frames initiate graceful shutdown. Control frames can be interleaved with data frames without breaking message reassembly.

## JPEG Encoder: Baseline ITU-T T.81

**File:** `image/jpeg_encoder.h`

JPEG encoding pipeline for screen capture compression:

```
RGB pixels → 8×8 blocks → color space → DCT → quantization → zigzag → Huffman → JFIF
```

### Fast DCT

Uses the Arai–Agui–Nakajima algorithm — a scaled integer DCT that requires only 5 multiplications and 29 additions per 8-point transform (vs. 16 multiplications for naive DCT).

### Streaming Output

JPEG data is written through a callback function (`bool (*writer)(void* ctx, const UINT8* data, UINT32 len)`), enabling direct streaming to the WebSocket without buffering the entire compressed image in memory.

Quantization tables are quality-scaled (1-100) per JPEG Annex K. Standard Huffman tables from Annex K.3 (no custom table optimization — trades compression ratio for code simplicity).
